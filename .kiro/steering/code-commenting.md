# Code Commenting Standards

本文档定义 Go 与 Dart/Flutter 代码注释的统一规范，贯穿整个开发过程。

## Go 注释规范

适用范围：所有新增或修改的 Go 代码文件（`backend/**/*.go`）。

### 必须注释的元素

- **struct** 必须有注释，说明结构体用途。
- **struct 字段** 每个字段必须有注释，说明语义与约束（如是否敏感、单位、默认值）。
- **可导出方法** 必须有注释，至少包含：方法用途、输入参数、输出参数/返回值语义。
- **非导出方法** 必须有注释说明方法用途；复杂方法需补充主要参数说明。
- **关键逻辑** 必须加必要注释，说明"为什么这样做"，避免只写显而易见的"做了什么"。
- **常量** 必须有注释，说明业务语义或使用场景。
- **全局变量** 必须有注释，说明生命周期、初始化方式与作用。

### 风格要求

- 注释要简洁、准确、可维护，避免空话与重复代码字面含义。
- **敏感信息**（密码、token、密钥等）不得写入注释、日志或错误明文。
- 与现有导出符号命名保持一致，避免注释与实现语义不一致。

## Dart/Flutter 注释规范

适用范围：所有新增或修改的 Dart 代码文件（`frontend/**/*.dart`）。

### 必须注释的元素

- **class** 必须有注释，说明类职责与使用场景。
- **类字段** 必须有注释，说明语义、单位、默认值或约束。
- **公有方法** 必须有注释，至少包含：方法用途、输入参数、返回值/副作用说明。
- **私有方法** 必须有注释说明用途；复杂方法需补充主要参数与关键分支说明。
- **关键业务逻辑、状态转换、异步边界**（如 `Future`/`Stream`）必须添加必要注释，说明"为什么这样做"。
- **常量** 必须有注释，说明业务语义与取值来源。
- **全局变量**（含顶层变量、单例实例）必须有注释，说明生命周期与作用域。

### 风格要求

- **优先使用 Dart 文档注释**：公有 API 使用 `///`，复杂实现可配合行内 `//` 注释。
- 注释应简洁、准确、可维护，避免复述代码字面含义。
- **敏感信息**（密码、token、密钥等）不得写入注释、日志或错误明文。
- 注释语义必须与实现保持一致，变更代码时同步更新注释。

---

_注释是代码可维护性的基石，而非形式主义的负担_
