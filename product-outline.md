# LoomiDBX 数据库模拟数据生成工具-产品大纲

## 1. 为什么做这个产品

目标：这是一个本地化部署的数据库模拟/合成数据生成工具。它接入用户数据库，识别数据库 Schema（表、字段、关系等），设定模拟数据生成策略，生成出逼真的模拟数据并回填到数据库中。
价值：降低测试、演示、性能研究、算法研究场景中的“准备数据成本”，创造出符合真实世界特征的数据，应用在软件功能演示、开发、测试的各个领域。

## 2. 目标用户

- 市场、售前人员：快速生成"漂亮"的 Demo 数据用于产品的演示；
- 数据库开发，测试人员：数据库性能测试/压力压测；
- 数据分析人员：帮助进行算法研究与验证，构造贴近真实分布的训练数据；
- 开发人员：开发环境数据填充，满足开发所需；

## 3. 用户痛点

- 现有工具生成的数据“太假”，通常是随机字符或数字的拼接，难以满足软件“演示”的需要
- 现有工具生成的数据大多不支持”关系和约束”，例如外键关系、唯一性约束等，生成的数据无法满足实际开发、测试的需要
- 手动数据准备的成本高昂，需要付出大量的心智和劳力成本

## 4. 成功标准

- 功能：用户能直接连接各种主流数据库，设定数据生成规则，生成数据并回写到数据库中
- 数据：生成的数据符合“现实世界”的规则（而非随机字符的拼接，例如订单编号），满足数据“关系和约束”的要求
- 性能：高性能输出，单机PC能满足 10000 条/秒的生成速度（口径：不包含外部数据源调用、不包含计算字段，包含唯一性约束检查，采用每批事务）
- 体验：简单易用，只要有数据库相关知识即可容易上手

### 4.1 MVP 范围与决议口径（2026-04-13）

- AI/LLM 生成不纳入 MVP，相关能力仅做预留
- 计算字段在 MVP 阶段仅支持 SQL 表达式，Python 表达式延后
- 种子策略支持全局 seed，用于复现测试与问题定位
- 密钥管理策略采用系统密钥环 + 环境变量注入

## 5. 主要功能清单

- 支持主流数据库的连接：MySQL、Postgres、Oracle、MSSQL、Hive、SQLite、ClickHouse 等。未来也支持云端厂商如 Amazon AWS, Google Cloud, Alibaba Cloud 等提供的云数据库
- 扫描并识别数据库 Schema，包括表、字段、关系、约束等
- 为每个表指定生成规则：生成数量、生成数据前是否先执行 Truncate 等
- 为每个字段制定数据生成策略：随机数/随机字符、满足概率分布、来自外部数据源、枚举值、计算值等（AI 生成为后续阶段能力）
- 生成数据后回写到数据库中，支持海量数据高速生成
- 保存数据库连接、Schema、表/字段生成规则并可复用
- 数据库表结构修改后，可重新扫描并映射规则

## 6. 核心卖点

### 6.1 生成"逼真多样"的数据

例如：

- **姓名**： 按照百家姓中的“姓”+中国人常见的“名”组成的姓名组合；
- **薪水**：满足帕累托分布或对数正态分布的薪资值
- **身份证**：满足 GB11643-1999 标准的身份证号码
- **地址**：按照中国大陆省市区县+街道地址合成
- **订单编号**：按照“固定前缀”+“年月日”+“流水号”规则生成
- **仓库名称**：由另一个系统（如 CRM 系统）的 API 提供
- **商品 ID**：来自维度表（商品表）的定义
- **支付状态**：枚举值（待支付、已支付、已退款），支持按权重分布
- **食品名称**：来自网络公开数据源（真实世界的数据）
- **电话号码**：由正则表达式生成
- **电影名称**：由 AI 根据真实电影市场随机生成（后续版本能力，非 MVP）

### 6.2 满足数据间的“外键关系”和约束

例如，生成订单数据时，其商品 ID 来自于商品表。
- 支持数据库中外键关系，也支持“逻辑外键”（手动指定表关系）
- 支持 1 对多、1 对 1 以及多对多关系（需要单独的关系表定义）

约束示例：

- 商品表中的“商品编码”字段满足“唯一性”约束
- 订单表中的“支付方式”字段来自一个枚举值（或数据字典），且不允许为 null 

## 7. 技术实现要点

系统分为前后端两个部分。

### 7.1 后端技术实现

后端使用 Go 语言开发：

1. 基于 database/sql 标准包连接各种主流数据库
2. 根据不同数据库，自动识别数据库 Schema 定义
3. 不同数据库的数据类型抽象化为 int, string, decimal, datetime, boolean 5 种类型
4. 每种数据类型可以拥有多种不同的“数据生成器”供用户选择，例如：int 类型可以有”序列生成器“、”区间均匀分布生成器“、”概率分布生成器“等
5. 每种生成器拥有不同的配置，例如”序列生成器“需要定义起始值、步长、字段约束（是否允许 Null、唯一性要求）
6. 支持”外键约束“以保证数据一致性，这种“外键约束”可以是物理外键（FOREIGN KEY），也可以是逻辑外键：由用户指定关联性
7. 支持数据库函数和表达式作为生成规则，例如一个订单号的定义可以是：`CONCAT('No', DATE_FORMAT(order_time, '%Y%m%d'), LPAD(seq, 3, '0'))`, 它基于一个表达式以及两个已经存在的列： order_time 和 seq
8. 支持格式化、整数截断。例如 5000-20000 之间的薪水值，最终生成的数据是”$12,500“（货币前缀, 千分位，按百分位截断）
9. 支持“维度表”：例如订单的“支付方式”字段来源于一个字典表，它可以通过类似 `select value from dict where key = 'pay_model'`这样的 SQL 语句抽取数据；也可以通过一个 JSON 文件来供给。甚至，可以通过 REST API 从另一个系统中获取
10. 使用 goroutine 提高并发性能
11. 后端代码通过 CGo 编译为动态库

### 7.2 前端技术实现

1. 前端使用 Flutter 作为 UI 框架层
2. 支持 Windows、MacOS 及 Linux
3. 界面风格上，满足“工具”属性，类似于 Navicat、DBeaver 等数据库管理工具，借鉴成熟的“数据库工具”界面风格，以减少使用人员的认知障碍
4. Flutter/Dart 通过 `dart:ffi` 直接调用后端提供的动态库
5. 对于 Dart FFI 处理 Go Struct 的方案，将使用 JSON 序列化的方式处理（Go 端使用 json-iterator），牺牲少量性能换取开发便利性

## 8. 主要使用场景及流程

### 8.1 用户初始进入后，创建一个数据库连接

1. 初始进入时，尚没有没有任何数据库连接及Schema，于是用户点击界面左侧面板上的“新建连接”按钮，从弹出的菜单中选择一个数据库或云数据库类型，如 MySQL、Postgres 等
2. 从弹出的对话框中填写连接参数：Host、Port、Username、Password及连接名称等（不同数据库类型有不同的连接参数）
3. 点击“测试连接”按钮测试通过后，点击“确定”保存
4. 界面左侧面板上展示一个 Tree 组件，显示刚刚创建的连接

### 8.2 用户显示某连接之下的所有数据库、表

1. 用户双击左侧面板上 Tree 组件上的某个连接，这个连接展开并显示其下的数据库
2. 用户可以进一步展开这个数据库，显示其下的所有表或模式（即 Schema，只针对Oracle 或 Postgres） 
3. 自动识别所有表及字段，扫描结果需持久化
4. 如果表的结构有变化，允许再次扫描

### 8.3 显示表及字段详情

1. 用户双击左侧面板上的某个表，在右侧面板上显示这个表的详情
2. 表的详情包含上下两个部分：
   - 上半部分显示表名称、注释、表的生成选项（生成数量、是否先执行 Truncate 等）、关联的表
   - 下半部分再次划分为左右两个部分，左侧显示表中的所有字段（名称、数据类型、是否主键、是否外键、如果是外键，其关联的表+字段、生成器名称），右侧是字段对应的“生成器设置选项”
3. 当表的结构发生了变化，再次扫描后，如果将影响原有生成器的设置，需要标注字段并允许用户调整生成器设置。例如增加了字段，或者修改了数据类型等

### 8.4 显示/设定每个字段的“生成器选项”

1. 用户点击某个字段，在右侧显示其对应的“生成器设置选项”
2. 用户可以设置/调整生成器的类型及选项。例如指定某个整数字段的生成器为“序列增长生成器”，其起始值为 1000，步长为 1
   - 不同数据类型的字段有不同的生成器可供选择，每种不同的生成器有不同的设置选项
   - 如果字段是外键字段，则不能指定其生成器，它的值只能来源于其参照表（Reference）中的字段
   - 可以根据字段约束设定选项：是否允许为 null，null 值的比例，是否是唯一值等
   - 对于 Postgress 或 Hive 数据库，允许它的值可以是一个数组
3. 允许为某个字段配置“逻辑外键”，逻辑外键与物理外键的作用相同
4. 允许某个字段不参与生成数据，但要求这个字段允许 Null 值，或存在缺省值
5. 允许某个字段是一个“计算”字段。MVP 阶段支持 SQL 表达式；Python 表达式能力已完成研究并作为后续阶段能力纳入规划（延期原因是阶段复杂度控制，而非方案不可行）。计算字段可以引用其它非计算字段的值
6. 可以使用“预览”功能对生成器的设定进行验证
7. 一旦设定完成，将永久保存用户的设置。默认情况下，所有配置保存在 SQLite 中，但用户也可以指定一个自己的数据库（MySQL/Postgres/Oracle/MSSQL 及协议兼容的数据库）

### 8.5 生成数据

用户可以在左侧面板上点击“生成数据”按钮生成数据。

1. 弹出“生成向导”对话框，列举当前数据库下的所有表。用户也可以选择另一个连接/数据库
2. 用户选择要生成数据的表，未设置生成选项的表不能被选中（用显著标识区分）
3. 用户为每个表设定要生成的数量
   - 每个表有一个默认生成数量，即设计生成器时指定的生成数量
   - 如果表存在外键关联，且关系为 1 对 0-1 ，则此表生成的数量与关联表生成的数量一致。例如发货单与订单，需要保证「每订单至多一发货单」
   - 如果表存在外键关联，且关系为 1 对多，则需要指定与关联表生成的数量的倍数关系。例如订单明细的记录数与订单主表的记录数在 1-30 倍之间
   - 数据表存在外键关联，但没有设定“生成关联表数据”，则此表也不能生成数据
   - 数据表存在外键关联，生成数据时，被依赖的关联表将先生成数据
4. 用户点击“生成”，开始生成并插入数据
5. 在插入完成前，一直显示进度对话框直至完成，最后显示完成情况：用时、每个表的最终插入记录数
6. 记录本次生成数据的情况，备查
7. 数据写入失败时，立即停止，且回滚当前失败批次事务。数据将按“批”写入数据库，每一个批就是一个事务（每批事务为默认基线），用户可以设定批的大小，例如 10000
8. 每个表插入数据前，可以先执行 Truncate（依据用户设定）
